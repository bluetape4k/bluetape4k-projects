# exposed-trino 모듈 설계 Spec

**날짜**: 2026-04-03
**대상**: `bluetape4k-projects/data/exposed-trino/` (신규)
**연관**: `data/exposed-bigquery/` (기존, 의존 관계 변경 예정)
**참고 패턴**: `data/exposed-duckdb/` (커스텀 Dialect + JDBC 연동)

---

## 1. 개요

### 1.1 배경

현재 `exposed-bigquery`는 H2(PostgreSQL 모드)로 SQL을 생성한 뒤 BigQuery REST API로 실행하는 구조이다.
이 접근은 다음 한계가 있다:

- H2 PostgreSQL 모드의 SQL 생성이 BigQuery/Trino SQL과 완전히 호환되지 않아 `toBigQueryDdl()` 같은 변환이 필요
- JDBC 트랜잭션이 없어 Exposed의 표준 `transaction {}` / `SchemaUtils` 활용 불가
- 타입 매핑을 수동으로 처리해야 함 (`BigQueryResultRow.convertValue`)

### 1.2 목표

- **`exposed-trino`**: Trino JDBC를 통한 범용 Exposed Dialect 모듈. Trino가 지원하는 모든 커넥터(Memory, Hive, Iceberg, Delta Lake 등)에 사용 가능
- **`exposed-bigquery` 개선**: H2 sqlGenDb를 Trino JDBC로 대체하여 BigQuery 커넥터를 통한 네이티브 실행

### 1.3 모듈 관계

```
exposed-trino              ← Phase 1 신규: TrinoDialect + TrinoDatabase + suspend/Flow API
exposed-bigquery           ← 기존: BigQuery REST API (독립 유지, 변경 없음)
exposed-bigquery-trino     ← Phase 2 신규: BigQuery + Trino 통합
    ├── implementation(exposed-bigquery)
    └── implementation(exposed-trino)
```

## 2. 설계 결정

### D1: Dialect 기반 클래스 — `PostgreSQLDialect`

Trino SQL은 PostgreSQL과 높은 호환성을 보인다 (ANSI SQL 기반).
DuckDB, BigQuery 모듈과 동일하게 `PostgreSQLDialect`를 상속한다.

**근거**: DuckDB(`DuckDBDialect : PostgreSQLDialect`), BigQuery(`BigQueryDialect : PostgreSQLDialect`) 모두 같은 패턴.

**상속 내러티브**: `TrinoDialect`는 **구조적으로는 DuckDB 패턴**을 따른다 (커스텀 Dialect + Metadata + ConnectionWrapper + Database 팩토리). **capability 플래그 설정은 BigQuery 패턴**을 따른다 (`supportsColumnTypeChange = false`, `modifyColumn()` no-op 등).

### D2: 표준 JDBC 방식 — DuckDB 패턴 따름 (비트랜잭션 autocommit 세션)

Trino는 공식 JDBC 드라이버(`io.trino:trino-jdbc`)를 제공하므로 BigQuery의 REST API 방식이 아닌 DuckDB의 표준 JDBC 방식을 따른다.

- `Database.connect()` → `transaction {}` → Exposed DSL 전체 활용 가능
- **단, `transaction {}`은 비트랜잭션 autocommit 세션이며 원자성이 보장되지 않음** (D8 참고)
- `SchemaUtils.create/drop` DDL 정상 동작
- ResultSet 기반 타입 매핑 자동 처리

### D3: TrinoDatabase 팩토리 — 연결 유형 다양화

Trino는 다양한 카탈로그/스키마를 지원하므로 팩토리에 카탈로그/스키마를 명시적으로 받는다.

**Phase 1 API** (이번 스펙):
```kotlin
object TrinoDatabase {
    fun connect(host: String, port: Int, catalog: String, schema: String, user: String = "trino"): Database
    fun connect(jdbcUrl: String, user: String = "trino"): Database
    // connect(dataSource) 는 Phase 2에서 제공 (TrinoDataSourceWrapper 구현 후)
}
```

> **결정**: `connect(dataSource)` 오버로드는 Phase 1에서 제외한다.
> Phase 1에서는 `connect(host, ...)` 와 `connect(jdbcUrl, ...)` 만 제공하며,
> 모든 Connection이 동일한 `TrinoConnectionWrapper` 런타임 계약을 갖는 것이 보장된다.
> HikariCP 등 커넥션 풀 지원은 Phase 2에서 `TrinoDataSourceWrapper` 구현과 함께 제공한다.

### D4: suspend + Flow 지원 — DuckDB 패턴 재사용

`suspendTransaction()`, `queryFlow()` 확장 함수를 동일 패턴으로 제공한다.

### D5: TrinoDialectMetadata — FK/제약 조건 우회

Trino는 FK, 인덱스 등 일부 JDBC 메타데이터를 지원하지 않는다.
DuckDB의 `DuckDBDialectMetadata` 패턴으로 `fillConstraintCacheForTables`를 no-op 처리한다.

### D6: exposed-bigquery 통합 — 2단계 접근 + 모듈 분리

**Phase 1** (이번 스펙): `exposed-trino`를 독립 모듈로 완성. `exposed-bigquery` 변경 없음.
**Phase 2** (후속): `exposed-bigquery-trino` **별도 모듈**로 통합 기능 제공.
  - `exposed-bigquery`는 REST API 모드 독립 유지 (변경 없음)
  - `exposed-bigquery-trino`에 `BigQueryTrinoContext.create(trinoHost, trinoPort)` 구현
  - `compileOnly` 트랩 완전 제거 — `implementation`으로 명시적 의존
  - REST API 모드는 deprecated 하지 않음 (Trino 없이도 사용 가능해야 함)

### D7: Trino 미지원 기능 처리 — 어노테이션 + 런타임 예외

```kotlin
@Target(AnnotationTarget.FUNCTION, AnnotationTarget.CLASS, AnnotationTarget.PROPERTY)
@Retention(AnnotationRetention.RUNTIME)
annotation class TrinoUnsupported(val reason: String = "")
```

- DDL 제한: `ALTER TABLE ADD/DROP COLUMN` 커넥터별 상이 → `supportsColumnTypeChange = false`
- AUTO_INCREMENT 미지원 → `supportsMultipleGeneratedKeys = false`
- 트랜잭션 미지원 (대부분 커넥터) → `autoCommit = true` 기본 설정

### D8: autoCommit 전략 — `TrinoConnectionWrapper` (비트랜잭션 autocommit 세션)

Trino Memory connector는 `BEGIN`/`COMMIT`/`ROLLBACK` 미지원이므로 Exposed `transaction {}` 블록이 런타임에 실패한다.
DuckDB의 `DuckDBConnectionWrapper` 패턴(`data/exposed-duckdb/src/main/kotlin/io/bluetape4k/exposed/duckdb/DuckDBConnectionWrapper.kt`)을 참고하여 `TrinoConnectionWrapper`를 추가한다.

**결정: B안 — `transaction {}` 유지, autocommit 계약 명문화**

Exposed의 `transaction {}` 블록을 그대로 노출하되, 다음 계약을 명시한다:
- Trino는 트랜잭션을 지원하지 않으므로 **autocommit 모드**로 실행된다
- 블록 중간 실패 시 이미 실행된 DML은 **롤백되지 않는다** (원자성 없음)
- `TrinoConnectionWrapper`의 `commit()`/`rollback()` no-op은 "예외를 숨기는 게 아니라 Exposed 프레임워크 호환을 위한 어댑터"이다
- `transaction {}` 블록은 실질적으로 **비트랜잭션 autocommit 세션**이며, multi-statement 쓰기 시 부분 반영에 주의해야 한다

**핵심 동작**:
- `autoCommit = true` 강제 설정
- `commit()` / `rollback()` 호출 시 no-op 처리 (Exposed 프레임워크 호환 어댑터)
- `prepareStatement(sql, autoGeneratedKeys)` 등 Trino JDBC 미지원 오버로드를 `prepareStatement(sql)`로 위임

```kotlin
/**
 * Trino JDBC Connection 래퍼. Exposed 프레임워크 호환을 위한 어댑터.
 *
 * ⚠️ **Trino autocommit 주의사항**
 *
 * Trino는 트랜잭션을 지원하지 않습니다. 이 블록은 **autocommit 모드**로 실행됩니다:
 * - 원자성(Atomicity) **보장 없음** — 블록 중간 실패 시 앞선 DML은 롤백되지 않습니다
 * - Rollback **불가** — `rollback()`은 no-op으로 동작합니다
 * - Nested transaction / Savepoint **미지원**
 * - Multi-statement 쓰기 블록 사용 시 **부분 반영 위험** 인지 필요
 *
 * `commit()`/`rollback()`은 예외를 숨기는 것이 아니라,
 * Trino의 autocommit 방식과 Exposed 인터페이스를 연결하는 **어댑터**입니다.
 * Exposed 프레임워크의 `transaction {}` 블록이 정상적으로 실행 흐름을 이어갈 수 있도록
 * 프레임워크 호환성을 제공하는 역할만 합니다.
 */
@Suppress("JavaDefaultMethodsNotOverriddenByDelegation")
internal class TrinoConnectionWrapper(private val conn: Connection) : Connection by conn {

    override fun getAutoCommit(): Boolean = true
    override fun setAutoCommit(autoCommit: Boolean) { /* no-op: Trino는 항상 autoCommit */ }
    override fun commit() { /* no-op: Exposed 프레임워크 호환 어댑터 — Trino는 트랜잭션 미지원 */ }
    override fun rollback() { /* no-op: Exposed 프레임워크 호환 어댑터 — Trino는 트랜잭션 미지원 */ }

    override fun prepareStatement(sql: String, autoGeneratedKeys: Int): PreparedStatement =
        conn.prepareStatement(sql)

    override fun prepareStatement(sql: String, columnIndexes: IntArray): PreparedStatement =
        conn.prepareStatement(sql)

    override fun prepareStatement(sql: String, columnNames: Array<out String>): PreparedStatement =
        conn.prepareStatement(sql)
}
```

**Phase 1 `TrinoDatabase.connect()` 오버로드에서 `TrinoConnectionWrapper` 강제 적용**:
- `connect(host, port, catalog, schema, user)`: `DriverManager.getConnection()` 결과를 `TrinoConnectionWrapper`로 감싼다
- `connect(jdbcUrl, user)`: 동일
- `connect(dataSource)`: **Phase 1에서 제외** — `TrinoDataSourceWrapper` 구현 후 Phase 2에서 제공. Phase 1에서는 `connect(jdbcUrl)` 또는 `connect(host, ...)` 만 사용하며, 모든 Connection이 동일한 런타임 계약을 갖는 것이 보장된다.

```kotlin
package io.bluetape4k.exposed.trino.dialect

class TrinoDialect : PostgreSQLDialect(name = dialectName) {
    companion object : KLogging() {
        const val dialectName: String = "trino"
    }

    // Trino는 ALTER COLUMN TYPE 미지원 (커넥터 의존)
    override val supportsColumnTypeChange: Boolean = false

    // Trino는 SERIAL/시퀀스 기반 자동 증가 미지원
    override val supportsMultipleGeneratedKeys: Boolean = false

    // Trino는 WINDOW FRAME GROUPS 지원
    override val supportsWindowFrameGroupsMode: Boolean = true

    // Trino는 ALTER TABLE MODIFY COLUMN 미지원 → no-op (BigQueryDialect 동일 패턴)
    @OptIn(InternalApi::class)
    override fun modifyColumn(column: Column<*>, columnDiff: ColumnDiff): List<String> = emptyList()
}
```

### 4.4 TrinoExtensions

```kotlin
package io.bluetape4k.exposed.trino

/**
 * Trino에서 suspend 트랜잭션을 실행합니다.
 *
 * ⚠️ **Trino autocommit 주의사항**
 *
 * Trino는 트랜잭션을 지원하지 않습니다. 이 블록은 **autocommit 모드**로 실행됩니다:
 * - 원자성(Atomicity) **보장 없음** — 블록 중간 실패 시 앞선 DML은 롤백되지 않습니다
 * - Rollback **불가** — `rollback()`은 no-op으로 동작합니다
 * - Nested transaction / Savepoint **미지원**
 * - Multi-statement 쓰기 블록 사용 시 **부분 반영 위험** 인지 필요
 */
suspend fun <T> suspendTransaction(
    db: Database,
    dispatcher: CoroutineDispatcher = Dispatchers.IO,
    block: Transaction.() -> T,
): T = withContext(dispatcher) {
    transaction(db) { block() }
}

/**
 * Trino 쿼리 결과를 Flow로 반환합니다.
 * JDBC ResultSet을 트랜잭션 내에서 List로 materialize한 뒤 순차 emit합니다.
 */
fun <T> queryFlow(
    db: Database,
    dispatcher: CoroutineDispatcher = Dispatchers.IO,
    block: Transaction.() -> Iterable<T>,
): Flow<T> = flow {
    val items = withContext(dispatcher) { transaction(db) { block().toList() } }
    items.forEach { emit(it) }
}
```

### 6.3 테스트 시나리오

| 테스트 | 카탈로그 | 검증 대상 |
|--------|----------|-----------|
| `TrinoDatabaseTest` | memory | Dialect 등록, 연결 성공, 기본 메타데이터 |
| `TrinoConnectionWrapperTest` | memory (실제 연결) | autoCommit 강제 true, setAutoCommit no-op, commit/rollback no-op, prepareStatement 위임, nested transaction 미지원 확인 |
| `TrinoTransactionAtomicityTest` | memory | autocommit 모드에서 부분 반영 동작 검증 — 블록 중간 실패 시 앞선 INSERT가 **롤백되지 않음**을 명시적으로 테스트 |
| `SchemaUtilsTest` | memory | `SchemaUtils.create(Events)` + `SchemaUtils.drop(Events)` DDL 동작 |
| `SelectTest` | memory | selectAll, where, orderBy, limit, count, groupBy |
| `InsertTest` | memory | insert, batchInsert |
| `TrinoExtensionsTest` | memory | suspendTransaction, queryFlow |

**테스트 상세 — `TrinoConnectionWrapperTest`**:
- `getAutoCommit()` 반환값이 항상 `true`임을 검증
- `setAutoCommit(false)` 호출 시 no-op (예외 없이 무시) 검증
- `commit()` 호출 시 no-op (예외 없이 무시) 검증
- `rollback()` 호출 시 no-op (예외 없이 무시) 검증
- `prepareStatement(sql, autoGeneratedKeys)` 호출 시 `prepareStatement(sql)` 위임 검증
- nested transaction 미지원: Exposed `transaction { transaction { } }` 패턴이 예외 없이 autocommit 세션으로 동작함을 확인

**테스트 상세 — `TrinoTransactionAtomicityTest`** (**필수 확정**):
- `transaction {}` 블록 내에서 INSERT 1건 실행 후 의도적 예외 발생 → 해당 INSERT가 DB에 **남아있음(롤백 안 됨)**을 검증
- 정상 흐름: INSERT → INSERT → 성공 → 양쪽 모두 반영됨 검증 (비교 베이스라인)
- nested transaction 시도 시 동작 확인: Exposed `transaction { transaction { } }` 패턴이 예외 없이 autocommit 세션으로 동작함을 확인 (예외 발생 여부 명시)
- 목적: autocommit 모드의 부분 반영 위험을 테스트로 명문화하여 사용자에게 경고

### 6.4 Trino Memory 커넥터 제약

- DDL: `CREATE TABLE`, `DROP TABLE` 지원
- DML: `INSERT INTO`, `SELECT` 지원
- **미지원**: `UPDATE`, `DELETE`, FK, 인덱스
- `ALTER TABLE`: Memory 커넥터를 포함해 커넥터/DDL 하위 기능별 지원 여부가 다르므로 실제 동작을 커넥터별로 검증 필요 (`ALTER TABLE ADD COLUMN` 등 일부 지원 가능)
- 테스트는 INSERT + SELECT 위주로 작성

### 6.5 Trino JDBC `prepareStatement` 호환성

Trino JDBC 드라이버는 `prepareStatement(sql, autoGeneratedKeys)` 등의 오버로드를 지원하지 않을 수 있다.
DuckDB와 동일하게 `TrinoConnectionWrapper`에서 이를 `prepareStatement(sql)`로 위임하여 호환성을 보장한다.
구현 시 실제 Trino JDBC 버전에서 해당 오버로드의 동작을 검증해야 한다.

## 7. exposed-bigquery 통합 계획 (Phase 2)

### 7.1 현재 구조

```
BigQueryContext
  ├── sqlGenDb: Database (H2 PostgreSQL 모드) → SQL 생성 전용
  ├── bigquery: Bigquery (REST API 클라이언트) → 실행
  └── toBigQueryDdl() → H2 DDL → BigQuery DDL 변환
```

### 7.2 모듈 분리 전략 (변경)

기존 `compileOnly(exposed-trino)` 방식은 사용자가 의존성 누락 시 `ClassNotFoundException` 런타임 오류를 유발하는 트랩이므로 **별도 모듈로 분리**한다:

```
exposed-bigquery        ← BigQuery REST API (기존, 독립 유지, 변경 없음)
exposed-trino           ← Trino JDBC (Phase 1 신규, 독립)
exposed-bigquery-trino  ← BigQuery + Trino 통합 (Phase 2 신규)
    implementation(project(":bluetape4k-exposed-bigquery"))
    implementation(project(":bluetape4k-exposed-trino"))
```

- `BigQueryTrinoContext`는 `exposed-bigquery-trino` 모듈에 위치
- `exposed-bigquery`와 `exposed-trino`는 서로 의존하지 않음 (완전 독립)
- `compileOnly` 트랩 완전 제거 — 사용자는 필요한 모듈만 `implementation`으로 추가

### 7.3 Trino 모드 구조

```
BigQueryTrinoContext  (exposed-bigquery-trino 모듈)
  ├── db: Database (Trino JDBC, bigquery 카탈로그) → SQL 생성 + 실행 통합
  └── Exposed 표준 transaction {} 그대로 사용 (비트랜잭션 autocommit 세션)
```

> **⚠️ 중요: `BigQueryTrinoContext`는 기존 `BigQueryContext`와 동치가 아니다.**
>
> Trino BigQuery 커넥터는 BigQuery REST API를 통해 접근하지만,
> `BigQueryContext`(REST API 직접 호출)와 비교해 다음 기능 차이가 있다:
>
> - **view 읽기 기본 비활성화**: `views-enabled=true` 커넥터 설정 없이는 BigQuery View를 조회할 수 없음
> - **단일 GCP 프로젝트 제한**: 커넥터 설정에서 지정한 하나의 GCP 프로젝트만 접근 가능; 다중 프로젝트 쿼리 불가
> - **passthrough query 읽기 전용 권장**: Trino BigQuery 커넥터의 native query pass-through는 읽기 전용 쿼리에만 사용 권장; 쓰기 DML 동작은 커넥터 버전별로 다를 수 있음
> - **일부 BigQuery 고급 기능 미지원**: INFORMATION_SCHEMA 일부, BigQuery ML, BI Engine 등

### 7.3.1 기존 BigQueryContext 사용자 마이그레이션 주의사항

`BigQueryContext` → `BigQueryTrinoContext` 전환 시 반드시 아래 사항을 확인하라:

| 항목 | BigQueryContext (REST API) | BigQueryTrinoContext (Trino 커넥터) |
|------|--------------------------|--------------------------------------|
| View 조회 | 기본 지원 | 커넥터 설정 `views-enabled=true` 필요 |
| 다중 GCP 프로젝트 | 가능 | 단일 프로젝트만 가능 |
| DML (INSERT/UPDATE) | REST API 직접 실행 | Trino 커넥터 DML 지원 범위에 의존 |
| DDL (CREATE TABLE) | `toBigQueryDdl()` 변환 필요 | TrinoDialect SQL 생성 후 Trino 실행 |
| 트랜잭션 원자성 | 없음 (BigQuery 자체 한계) | 없음 (autocommit 모드) |
| BigQuery ML / BI Engine | 지원 (REST API 통해) | 미지원 |

> **권장**: 기존 `BigQueryContext` 사용자는 위 제약을 검토한 후 전환 여부를 결정하라.
> Trino 인프라가 없거나 위 기능이 필요한 경우 `BigQueryContext` (REST API 모드) 유지 권장.

3. 장기적으로 REST API 모드 유지 (Trino 없는 환경 지원)

## 8. Trino SQL vs PostgreSQL 차이점 대응

| 항목 | PostgreSQL | Trino | TrinoDialect 대응 |
|------|-----------|-------|-------------------|
| AUTO_INCREMENT | SERIAL/BIGSERIAL | 미지원 | `supportsMultipleGeneratedKeys = false` |
| UPDATE | 지원 | 커넥터별 상이 | Memory 커넥터 미지원, 별도 처리 불필요 |
| DELETE | 지원 | 커넥터별 상이 | Memory 커넥터 미지원 |
| ALTER TABLE | 지원 | 커넥터/DDL 하위 기능별 검증 필요 | `supportsColumnTypeChange = false` |
| 트랜잭션 | MVCC | 커넥터별 상이 | autoCommit 기본 |
| BOOLEAN | TRUE/FALSE | TRUE/FALSE | 호환 |
| VARCHAR | VARCHAR(n) | VARCHAR | length 제한 무시 가능 |
| TIMESTAMP | TIMESTAMP | TIMESTAMP | 호환 |
| DECIMAL | DECIMAL(p,s) | DECIMAL(p,s) | 호환 |
